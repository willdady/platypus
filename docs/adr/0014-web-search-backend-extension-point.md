# Web-search backends are a fourth Extension point routed through the search toggle

Platypus gains a fourth core-owned Extension point — a **Web-search backend** — that fills the request-gated "web search" chat toggle for Providers without working native search (self-hosted OpenAI-compatible servers: vLLM, LiteLLM, SGLang…). A Web-search backend is a Plugin Contribution (ADR-0013) that supplies a small, fixed pair of ephemeral tools — a mandatory `web_search` and an optional `read_url` — injected **only when the toggle is on**, exactly like today's native provider search, and gone when it is off. Selection is **per-Provider** (a nullable `webBackend` column), mirroring how native search is already resolved per-Provider; the backend's config/credentials live in deploy-time Plugin config (ADR-0013), not on the Provider row. This lets a self-hosted deployment get first-class web search + page reading — backed by whatever the Operator runs (e.g. SearXNG for search, a browser service for reading) — without baking vendor specifics into core and without the always-on tool-schema bloat of wiring those services as MCP servers. Originates from [#329](https://github.com/willdady/platypus/issues/329).

## Considered Options

- **A single Provider-config field (hardcode one blessed search URL in the OpenAI-compatible branch).** Rejected: lightweight (one field, one branch) but loses swappability — every new backend (Brave, Tavily, a different self-hosted stack) is another core change, and it bakes one vendor's specifics into core. Sandbox backends (ADR-0001/0013) already set the pluggable precedent for a "swappable, deployment-chosen capability"; a config field is the less principled shape given that precedent exists.
- **Connect SearXNG / a browser service as MCP servers.** Rejected as the sanctioned path: works today with no core change, but MCP tools are **always-on** — injected into the tool schema every turn regardless of the toggle — which is the tool-schema bloat this point exists to avoid, and it has no tie-in to the request-gated toggle. MCP remains valid for anyone who just wants to wire a search service in without core support.
- **An always-on Plugin Tool set.** Rejected: minimises tool count vs full MCP but is still always-on and still cannot ride the ephemeral toggle — the defining trait of this capability.
- **Selection per-Workspace (mirror sandbox backends exactly).** Rejected in favour of per-Provider: native search is already resolved per-Provider, and a Web-search backend is precisely a _fallback for a Provider that lacks native search_, so it belongs next to the `nativeSearchEnabled` knob. Per-Workspace would split the "can this Provider search?" decision across two rows, forcing the toggle-visibility gate to fetch Workspace state. Per-Provider keeps that gate a pure function of the Provider. (The backend's config/credentials are deployment-wide in Plugin config regardless, so per-Workspace's "configure once" appeal is weaker than it looks.)
- **Backend contributes its own arbitrary named tool set.** Rejected: the Extension point defines **exactly** `web_search` and `read_url` and only those — no backend-invented tools. This is the ADR-0002 "fixed core" philosophy applied to the toggle slot: the model's behaviour stays backend-agnostic and swap-safe, so an Agent prompt is not coupled to which backend an Operator configured.
- **Both tools mandatory.** Rejected: `read_url` is **optional**. A search-only Operator (SearXNG, no browser service) returns just `web_search`; when `read_url` is absent the model only gets search that turn. Only `web_search` is mandatory — a Web-search backend that cannot search is meaningless.
- **Two independently-selected backends (a search slot and a read slot).** Rejected: more flexible but pushes selection from one id to a pair of slots. One backend wraps both services (their split is its internal detail), preserving the single per-Provider selection id.
- **Native-search-first resolution (`native ?? webBackend`, as the originating issue proposed).** Rejected: the OpenAI-compatible branch of `openProvider` returns a (dead) OpenAI-hosted `web_search` even for a vLLM endpoint that cannot service it, so a native-first `??` short-circuits and the configured backend is **never reached** — on exactly the Provider this feature targets. Resolution is therefore **explicit-plugin-first**: if a `webBackend` is set, use it; native search is the fallback. Explicit Operator selection beats implicit capability, and this needs no change to the existing native-search wiring.
- **Carry `userEmail` on the backend context.** Rejected for v1: the context mirrors `SandboxContext` — `{ orgId, workspaceId, userId }` — where `userId` covers audit/attribution. `userEmail` (PII) is deferred to a separate enhancement if a backend genuinely needs per-user upstream identity; adding a context field is append-only, so nothing is lost by waiting, and it would then apply to `SandboxContext` too, keeping the two shapes uniform.
- **A signed per-user identity token into the backend.** Rejected as a core concern: Plugins run in-process with no isolation (ADR-0013), so core hands the backend plain context values via a trusted in-process call. If a backend chooses to attribute/authenticate to its _own_ upstream service, forwarding identity (signed or otherwise) is the backend's implementation detail, using its own Plugin credentials.

## Consequences

- **New Contribution type.** `packages/plugin-sdk` gains an optional `webBackends?: WebBackendContribution[]` key on `PluginContributions` (append-only, minor API bump per ADR-0013 — disturbs no existing Plugin). A `WebBackendContribution` carries a `backend` discriminator (auto-namespaced for third parties, flat for core, per ADR-0013), a display `name`, optional `configSchema` / `credentialsSchema`, and a `createTools(ctx, plugin?)` factory returning `{ web_search, read_url? }`. The factory shape parallels `SandboxBackendContribution.create` and `ToolSetTools` so it stays forward-compatible.
- **Backend context.** `WebBackendContext = { orgId, workspaceId, userId }`, mirroring `SandboxContext`. No `userEmail`, no signed token; a plain in-process call.
- **Registry & loader.** A `webBackends` registry with the same id-namespacing and boot-time collision handling as sandbox backends; the loader drives registration from the manifest (Plugin authors never call the internal registrar). Boot posture is fail-loud, runtime resolution graceful — a Provider pointing at a since-disabled backend degrades to no tools that turn, as any absent backend does today.
- **Selection.** A nullable `webBackend` column on the Provider row, resolved against the registry at turn time. Config/credentials come from deploy-time Plugin config (ADR-0013), not the Provider row.
- **Turn-time resolution (explicit-plugin-first).** When the toggle is on for a turn, the search slot is filled by the resolved `webBackend`'s `createTools(...)` if one is set, otherwise by native provider search, otherwise nothing. Tools remain ephemeral and toggle-gated; the native-search path is otherwise unchanged.
- **Capability gate.** A positive `providerHasNativeSearch(provider)` predicate — a pure function of `providerType` + `apiMode` (Anthropic, Google, OpenRouter, and OpenAI not in chat-completions mode → yes; Bedrock and OpenAI in chat-completions mode → no) — replaces the hardcoded `!== "Bedrock"` exclusion. The toggle is offered when `nativeSearchEnabled !== false && (providerHasNativeSearch(provider) || Boolean(provider.webBackend))`, killing the current dead switch on no-native-search Providers. The **backend-side** injection gate applies the same capability logic, so a stale client sending `search: true` on a no-native-search Provider without a backend cannot inject the dead OpenAI-hosted tool. Accepted edge: OpenAI in chat-completions mode is treated as no-native-search — matching the existing documented assumption that chat mode is, in practice, only used against self-hosted OpenAI-compatible servers.
- **Boundary vs MCP (ADR-0013 non-goal #4).** A Plugin Tool set that merely proxies HTTP is an MCP smell. A Web-search backend is different on two counts: it fills core's own **request-gated toggle slot** — a slot no MCP or Tool set can fill — and it is the same category as sandbox backends, a swappable deployment-chosen capability, not a duplication of MCP's always-on per-Agent tool exposure.
- **Out of tree.** No SearXNG / browser-service code ships in core; those are out-of-tree Plugins. Core ships the Extension point, registry, selection, resolution, and gate only.
