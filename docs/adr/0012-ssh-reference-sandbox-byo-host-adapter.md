# SSH reference Sandbox adapter attaches to a pre-existing, operator-owned host

Platypus ships a second reference Sandbox adapter (`backend: "ssh"`) that connects to an already-running SSH server the operator supplies — bring-your-own-host — rather than provisioning any compute. One `sandbox` row maps to exactly one connection target: `config` carries `{ host, port, user, rootDir?, hostKey? }` and `credentials` carries `{ privateKey, passphrase? }` (public-key auth only). The adapter never creates or destroys the machine; it connects, ensures the workspace root exists, and runs the fixed five-tool core over the connection. It ships as the core plugin `@platypus/ssh`, contributing a Sandbox backend via the plugin model in [ADR-0013](0013-plugin-system-manifest-driven-in-process-extension-points.md); it is enabled by listing `@platypus/ssh` in `PLATYPUS_PLUGINS`, not by a dedicated env gate. **This ADR therefore depends on and is sequenced after ADR-0013** — only the registration/enablement mechanism changed; every backend decision below is unaffected. It is the recommended backend for horizontally-scaled deployments, where the Docker adapter cannot run (ADR-0003).

## Considered Options

- **SSH-then-provision** (SSH to a host and create a per-Workspace container / user / chroot, Platypus owning that sub-resource's lifecycle). Rejected for v1: pulls remote-Docker/systemd assumptions and a teardown-reconciliation story into an adapter whose whole appeal is "point it at a box you already have." Attach-only keeps the surface small; provisioning can be a distinct future backend.
- **Wipe `rootDir` on `destroy()`** (symmetric with the Docker adapter removing its volume). Rejected: the host is not Platypus-owned, so a mis-configured `rootDir` (`/`, a shared home) would delete operator data. `destroy()` is a no-op (disconnect only); remote filesystem hygiene is the operator's responsibility. The Docker asymmetry is deliberate, not an omission.
- **Password auth (or key-or-password union).** Rejected for v1: public-key is the standard for headless SSH, is easy to scope and rotate per host, and avoids storing a reusable password in plaintext. Hardened hosts often disable password auth anyway. Add later if a real host needs it.
- **`exec` for all five tools** (mirror the Docker adapter's shell path). Rejected: Docker's safety comes from `container.exec({ Cmd: [...] })` argv-arrays and from writing via `putArchive` — _not_ the shell. SSH `exec` is a single shell string with no argv form, so "mirror Docker" would mean hand-written path quoting and fragile base64-pipe writes with no atomic create — re-earning, buggily, a property Docker got for free. We use SFTP for `fs.read`/`fs.write`/`fs.edit` (the true `putArchive` analogue: literal paths, native `wx`=create / `w`=overwrite) and `exec` `find -printf` for `fs.list` (the `find` analogue, one-shot recursion). `shell.exec` is inherently a shell string and stays on `exec`.
- **SFTP for `fs.list` too** (client-side recursive walk). Rejected: deep recursive lists cost one round-trip per directory; `find -printf` returns the whole tree in one call, matching the Docker adapter's listing shape. The only injection surface is that single list path, and the input schema already rejects absolute paths.
- **Required host-key pin (always strict), or accept-any (no verification).** Rejected the extremes. Public-key auth already prevents an impostor host from stealing credentials or breaking into the real host, so the residual risk is session/output exposure (including the ADR-0004 injected env secrets) to a MITM. We take an _optional_ `hostKey` pin: present → strict verify, fail on mismatch; absent → connect with a loud warning. Frictionless on localhost/dev, lockable for internet-facing hosts.
- **Per-turn connection via a disposer wired into `chat-execution`'s `dispose()`.** Rejected for v1: the static tool-set resolver returns `Record<string, Tool>` with no disposer channel, and adding one is a cross-cutting contract change shared by every tool set for a per-adapter need. The connection is instead self-managed inside the adapter.

## Consequences

- New backend discriminator `"ssh"` (a core, unprefixed Contribution id per ADR-0013), contributed by the `@platypus/ssh` core plugin through `contributes.sandboxBackends` — _not_ a side-effect registration and _not_ gated by `PLATYPUS_SANDBOX_SSH_ENABLED`. Available only when `@platypus/ssh` is listed in `PLATYPUS_PLUGINS`; not listed ⇒ tools absent that turn, the same degradation as any absent backend. The per-Workspace `config`/`credentials` above stay per-Workspace Sandbox settings (ADR-0001/0006) — ADR-0013's deploy-time _plugin_ config does not apply to them.
- `config = { host, port, user, rootDir?, hostKey? }`, `credentials = { privateKey, passphrase? }`. `rootDir` is optional and defaults to `$HOME/platypus-workspace`, created with `mkdir -p` on connect; the adapter resolves `$HOME` once per connection so SFTP (which does not expand `~`) has an absolute root. As with every adapter the model only ever sees workspace-root-relative paths, so the physical root is free.
- The backend image gains the `ssh2` (npm) dependency for the connection, `exec`, and SFTP subsystem access.
- **Connection lifecycle is self-managed.** A backend instance is minted per Chat turn; it lazy-connects on the first tool call, reuses the single connection across all tool calls in the turn (inflight-promise pattern, as Docker shares one `ensureContainer` promise), and an `unref()`'d idle timer closes it after inactivity. No cross-turn pooling; no changes to the tool-set contract.
- **Field-level authorization (ADR-0006):** every SSH-specific field is execution-target- or reach-determining, so `host`, `port`, `user`, `rootDir`, `hostKey`, `credentials`, and `adminEnv` are admin-only; only `name` and `userEnv` stay Workspace-Owner-editable — the same split as the Docker backend. The Sandbox remains non-delegatable.
- `shell.exec` implementation notes: `cwd` is honoured by prefixing `cd <rootDir>/<cwd> && …` (no native `WorkingDir` over SSH); the ADR-0004 merged env is applied via `export KEY=val;` statements rather than the `exec` `env` option, since sshd's `AcceptEnv` rejects arbitrary variables by default. Output caps, truncation flags, and the 60s/600s timeout contract are unchanged from ADR-0002; timeout enforcement closes the channel and reports exit code 124 (a remote command may keep running as an orphan — the host is not ours to reap).
- **ADR-0003's resource limits do not apply.** There is no `PidsLimit`/`Memory`/`NanoCpus` equivalent; a fork bomb or memory hog on the BYO host is the operator's problem. Bounding it (remote `timeout`/`ulimit`/`systemd-run`) is deferred to the operator.
- **ADR-0005's reachability model does not apply, and its DB-unreachability property is lost.** There is no `networks`/`extraHosts` config; the Sandbox reaches whatever the host can route to. The "Platypus database remains unreachable" guarantee held only under Docker's sibling-bridge topology — a BYO host that can route to the database exposes it to the agent. Reach is entirely the operator's network posture.
- **Fills ADR-0003's horizontal-scale gap.** Because the connection target is external and shared, any backend pod can SSH to the same host and see the same filesystem state — so the SSH adapter is viable in the multi-node topology where the Docker adapter is not.
- Scoped, like the Docker adapter, to environments where the operator trusts the target host and the network path; it is not a hostile-tenant isolation boundary.
