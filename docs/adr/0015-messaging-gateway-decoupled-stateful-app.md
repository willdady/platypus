# The messaging Gateway is a decoupled, stateful app bridging chat Surfaces to a messaging-agnostic backend

Platypus can be reached from external chat **Surfaces** (Telegram, Slack, Discord, …) through a **Gateway** — a decoupled, stateful app deployed alongside the frontend and backend. The Gateway holds the long-lived per-Surface connections and relays messages **both ways**; the **backend stays messaging-agnostic** — inbound rides the existing Chat API, outbound reuses the webhook event bus's transport, and no backend code knows about any Surface. Channel integrations are **Gateway adapters** behind the Gateway's _own_ adapter seam — first-party and in-repo — **not** Contributions to the backend Plugin system (ADR-0013). **Platypus, not the Gateway, is the identity authority.** This **revises** the roadmap's "each channel is a plugin extension point, same pattern as Sandbox backends" and **supersedes** ADR-0013's note that a messaging-gateway adapter would be a third backend Extension point (ADR-0013's text is left unchanged; this ADR is the correction of record). Human-in-the-loop approvals are explicitly **out of scope** for the first version.

## Considered Options

- **Adapters as backend Plugins / a third backend Extension point (in-process).** Rejected. A Gateway adapter holds a **persistent Surface connection** — Discord's Gateway WebSocket is _mandatory_ (no HTTP-events alternative), Slack Socket Mode is an outbound WebSocket, Telegram long-polling is a single-consumer loop. That is a sticky, stateful runtime shape **incompatible with the stateless, recycle-safe backend**, and one the backend must never grow (roadmap non-goal: "no messaging/notification stack inside the backend"). A `toolSet`/`sandboxBackend` Plugin cannot run in the Gateway app and a Gateway adapter cannot run in the backend, so sharing ADR-0013's manifest/loader across the two apps is a false economy — the Gateway gets its own seam.

- **A thin relay holding sockets, with adapter logic still in the backend.** Rejected: smears per-Surface messaging code back into the backend (the non-goal) and makes "scale independently" illusory — the logic still lives in, and scales with, the backend.

- **A persistent bidirectional socket, or a message broker, between Gateway and backend.** Rejected for v1. A socket reintroduces the long-lived stateful coupling on the backend we are trying to keep out; a broker (Redis Streams/NATS) adds an operational dependency a self-hoster must run. **Plain HTTP both ways** suffices: inbound calls the Chat API and consumes the reply on that request's SSE response; outbound-push reuses the existing **HMAC-signed, retrying webhook bus transport**. Revisit a broker only when one Gateway instance provably can't cope.

- **Third-party adapter packages in v1.** **Deferred, not designed out.** Adapters are first-party and in-repo behind a uniform capability contract (auth / inbound / outbound / streaming / threading / pairing), à la OpenClaw's in-monorepo `ChannelPlugin` extensions. The seam is kept clean so external adapters — the long tail (Matrix, WhatsApp, regional apps) — become an additive change later. First-party authorship and a real seam are orthogonal; we take both.

- **The Gateway as an identity authority (mint sessions / impersonate Users).** Rejected — it would break multi-tenancy. Platypus owns identity via **two records**: an **Identity link** (`Sender → User`, _authorizes_) and a **Conversation binding** (`Conversation locus → Chat`, _routes_). They are separate because in a shared room "who spoke" and "where it happened" diverge; a DM merely collapses them 1:1. Linking is **UI-first**: a Platypus-minted, **single-use, short-TTL** code bound to the authenticated User + chosen Agent, relayed to the Surface (Telegram `?start=` deep link, or a `/link <code>` command). The Gateway _relays_ the code but cannot forge one, and can act only as **already-linked** Users. (OpenClaw's operator-approved allowlist is single-tenant; our multi-tenancy requires the User-authenticated variant — we borrow its concrete shape: ~8-char unambiguous alphabet, ~1h TTL, capped pending, pending-vs-durable records kept separate.) Independently, the Gateway verifies each Surface's **own** inbound signature (Slack signing secret, Discord Ed25519) — which proves _platform origin, not user identity_ — separately from the HMAC on the Gateway↔backend hop. Two checks, two jobs.

- **Routing in-app Notifications to Surfaces (the Surface as a second view of the notification centre).** Rejected. A Surface conversation is a **single linear stream**; an in-app Notification projected onto it appears inline as a message and **diverges from the Chat transcript** — and a Notification is Workspace/Agent-scoped, not Chat-scoped, so "which conversation" is undefined. Invariant instead: **a Conversation locus mirrors exactly one Chat and shows only that Chat's messages.** Agent-initiated (proactive) output is therefore a **Chat message appended to the bound Chat**, delivered over a new outbound event — _not_ the in-app Notification object, which stays in-app and is never routed to a Surface. Proactive delivery to a DM is **explicitly opted-in per target**: linking alone never authorizes an Agent to message a User (OpenClaw's "pairing ≠ automation recipient" safety rule).

- **Proactive messages start a _new_ conversation on every Surface.** Rejected: impossible on single-stream Surfaces — a Telegram bot DM has a **cardinality-one** locus, with nowhere to put a parallel conversation. Proactive output **appends** to the rolling Chat; "start a new conversation" is a **thread-capable-Surface affordance** (a new Slack/Discord thread → new binding → new Chat, auto-derived). On single-stream Surfaces, **`/new` rebinds** the one locus to a fresh Chat (old Chat preserved, a separator message marks the boundary). v1 constraint: **one bound Agent per single-stream DM**.

- **Human-in-the-loop approvals in v1.** Deferred. Approvals require **pause/resume in the run lifecycle** (`apps/backend/src/runs/`) — a larger, independent change than the Gateway itself. The contract is shaped so a later approval _request_ is an outbound Chat message with action affordances and a _decision_ is an inbound event — additive, no contract change.

- **Streaming tokens to the Surface (edit-in-place as the default).** Rejected for v1: per-message rate limits make token-rate edits stall on Slack/Discord. Default is **buffer-and-post-final with a typing indicator**; progressive edit-in-place is a **per-adapter opt-in** later (the SSE deltas are already available, so no core change enables it).

- **A multi-instance / sharded Gateway, or a forced-stateless webhook-only Gateway, in v1.** Rejected both. Sharding (shard ownership, leader election) is distributed-systems machinery for a scale almost no self-hoster reaches. Webhook-only ingress would drop **Discord entirely** (WS-mandatory) — the very Surface that forces the stateful design — and demands public ingress. v1 is a **single Gateway instance** sized by connection count; compute (Chat turns) stays in the backend and scales there. The split's purpose is **keeping sticky connection state out of the backend**, not autoscaling the Gateway like the backend.

## Consequences

- **Transport.** _Inbound:_ the Gateway calls the Chat API and relays the reply from that request's SSE stream (no bus). _Outbound proactive:_ the backend resolves the binding and emits a new **conversation-addressed outbound Chat-message event** (`{ surface, locus, chatId, content }`) reusing the webhook bus transport (HMAC, retry) — **distinct from `notification.*`, which the Gateway never consumes.** A synchronous reply and a proactive append are different paths because long-running work outlives any reasonable SSE lifetime. _Gateway↔Surface_ transport is per-adapter (Telegram poll/webhook, Slack Socket/HTTP, Discord WS).

- **Backend changes.** A way to construct the acting `Principal` (`User`) from a resolved **Identity link** via a **non-session path** (extending the existing `User | Trigger | SubAgent` seam); a new outbound Chat-message event type; storage for **Identity link** and **Conversation binding** records (Platypus owns identity, so these live in the backend DB); and the linking-code mint/validate. **No Surface-specific code enters the backend.**

- **Gateway app.** Its own adapter seam + registry; first-party in-repo adapters implementing the uniform capability contract; **Surface credentials (bot tokens) live in the Gateway**, not the backend, since the Gateway holds the connections.

- **v1 scope.** **Telegram adapter first** (native deep-link linking, no OAuth/ingress, exercises the single-stream / `/new`-rebind path we were least sure of), **Slack second** (validates the thread = Chat branch), **Discord later**. Deferred: HITL approvals, third-party adapters, progressive streaming, multi-instance/sharding.

- **Roadmap / ADR reconciliation.** Corrects three pieces of roadmap wording — (1) channels are a separate seam in a separate app, _not_ a backend Plugin Extension point; (2) "outbound rides the webhook event bus" is true only as _transport_ — channel content is Chat messages, not the Notification object; (3) "subsumes notifications and approvals" — the _capability_ is delivered via Chat messages, and approvals are deferred. Supersedes ADR-0013's planned "messaging-gateway adapter as a third Extension point" without altering ADR-0013's recorded history.
