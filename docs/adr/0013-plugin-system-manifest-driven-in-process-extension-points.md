# Plugins are manifest-driven, in-process bundles that fill core-owned extension points

Platypus gains a first-class way to extend itself **without maintaining a fork**: a _Plugin_ is a distributable package — one version, one config namespace, one enable/disable switch — that the Operator installs at deploy time and that core loads **in-process** (no isolation). Each Plugin exports a typed **manifest** whose `contributes` block declares one or more **Contributions** to a fixed set of core-owned **Extension points** (initially Sandbox backends and Tool sets; a messaging-gateway adapter is planned as a third). Core reads the manifest and drives registration itself — the existing `registerSandboxBackend` / `registerToolSet` functions become core-internal, no longer the plugin-facing API. **Everything is a Plugin, including core built-ins**, which ship pre-bundled under the `@platypus/*` naming convention and flow through the identical `contributes` path. Being a Plugin is about the _mechanism_ (manifest → `contributes` → register), not about being _optional_: the essential core tool sets load **always-on**, independent of the enable list (see the enablement option below), while the list gates only the plugins an Operator would plausibly deny. This settles the loading model, manifest format, and config mechanics the roadmap deferred to an ADR.

## Considered Options

- **Bare side-effect registration for external packages** (an out-of-tree package calls `registerSandboxBackend()` on import, as in-tree backends do today). Rejected: self-registration only works because core hard-codes `import "./sandbox/backends/index.ts"`; an external package can only self-register if _something_ imports it, and "no fork" forbids core growing an import line per plugin. The manifest inverts this — core discovers packages from an Operator-supplied list, `import()`s each, reads its manifest, and calls the internal `register*()`. Core stays ignorant of specific plugin names; the plugin author never calls `register*()`.
- **Pure-JSON `contributes` manifest (literal VS Code model).** Rejected: our Extension points need _code_ — a Sandbox backend ships a `create()` factory and two Zod schemas; a Tool set ships live `Tool` implementations. None survives as JSON. The manifest is a **typed TS object** exported from the package entry; only its top level (name / version / apiVersion / config namespace) is cheap declarative metadata. We borrow VS Code's `contributes` shape and its fixed contribution-point set — _not_ its marketplace, hot-loading, or sandboxed extension host.
- **A scanned `plugins/` directory** for discovery. Rejected in favour of an explicit Operator-supplied list (`PLATYPUS_PLUGINS`, promotable to a config file if it grows): the list _is_ the complete, auditable statement of what runs — read one value, know exactly what is active — and it doubles as the enable switch (below).
- **Flat, un-namespaced Contribution ids with boot-time collision detection.** Rejected _for third-party_: a third-party id could collide with a _future_ core built-in and break an existing deployment on upgrade. We take a **hybrid** — core Contributions keep flat ids (no data migration; every persisted `agent.toolSetIds` / `sandbox.backend` reference keeps working), while third-party Contributions are **auto-namespaced by plugin name** (`daytona.management`, `daytona.sandbox`). Authors write bare ids; core prefixes at load. The core-vs-third-party boundary is a **core-shipped allowlist of plugin names** (the packages this repo bundles) — _not_ the package-scope string, and _not_ list-membership (core plugins are listed too). `@platypus/*` is the naming convention for those packages, but the allowlist, resolved from the trusted bundle, is the authority: an Operator-installed `@platypus/impostor` is absent from the allowlist and is therefore treated as third-party (prefixed, curated-surface-only), so scope impersonation cannot smuggle a package in as core. This keeps the boundary — and the security model — independent of public-npm scope ownership.
- **Auto-namespace everything, including core.** Rejected: rewrites every stored id and forces a backfill across agents and sandboxes for no benefit the scope-based hybrid doesn't already deliver.
- **Per-workspace Plugin config, or Org-Admin-editable Plugin config in-app.** Rejected: contradicts "trust boundary is the deployment, no in-app install button." **Plugin config is deploy-time and Operator-owned**, keyed by plugin name (the "one config namespace"), validated at boot against a `configSchema` / `credentialsSchema` the manifest declares at the _plugin_ level. This is where a Plugin's **shared credential block** lives — one Daytona token injected into both its Sandbox backend and its management Tool set. The consequence — one credential block serves all tenants — is accepted; genuinely tenant-specific settings stay in the existing per-Workspace Sandbox row (ADR-0001) or an MCP. Plugin config is a _deployment_ layer above per-Workspace _resource_ config; the two layer, they do not merge (echoing the `adminEnv ⊃ userEnv` precedence of ADR-0004/0006).
- **A separate `_ENABLED` switch in addition to list membership.** Rejected: for a **gate-able** plugin, membership in `PLATYPUS_PLUGINS` _is_ the enable switch — one action, and disabling means removing the entry. Both changes need a backend restart anyway (deploy-time). This retires the per-backend env gates (`PLATYPUS_SANDBOX_DOCKER_ENABLED`, `PLATYPUS_SANDBOX_SSH_ENABLED` of ADR-0003/0012); Docker's opt-in posture falls out for free (just don't list it), so a manifest `enabledByDefault` field is unnecessary.
- **Gating _every_ plugin behind the list, core tool sets included.** Rejected. The list gates only plugins that pass the deny-in-isolation test (below); the essential core tool sets — `@platypus/tools-basic` and `@platypus/tools-platform` — load **always-on**, unconditionally, ahead of the list and independent of it. Gating them buys nothing (an Operator has no reason to deny math or a workspace's own kanban/memory) while inviting a sharp footgun: a hand-edited or clobbered list that drops them leaves every Agent silently tool-stripped — indistinguishable from data loss, the very failure the fail-loud boot posture exists to prevent. The always-on set is a core-shipped constant, a subset of the built-in map; the two remaining core plugins (`@platypus/web-fetch`, `@platypus/docker`) plus all third-party plugins stay gate-able. Consequences: listing an always-on plugin in `PLATYPUS_PLUGINS` is a **fail-loud misconfiguration** (it is not an enable switch), and an empty list no longer means "no tools" — it means "only the always-on core set." The always-on plugins remain full plugins (boot log, `GET /plugins`, catalog annotations); they are simply not _optional_.
- **One core plugin per Contribution, or one monolithic core bundle.** Rejected both. Core built-ins are split by the rule **"a capability gets its own plugin when an Operator would plausibly want to deny it in isolation"** — so `@platypus/web-fetch` (network egress) and `@platypus/docker` (infra) stand alone, while pure utilities (`@platypus/tools-basic`: math, time) and Platypus-domain tools (`@platypus/tools-platform`: kanban, dashboards, triggers, memory, notifications, agent-discovery, skill-/agent-management) group by cohesion. Egress-in-isolation matters specifically for the sovereignty-focused deployment. This same rule decides the always-on set: `web-fetch` and `docker` pass the deny-in-isolation test and stay gate-able, while nothing in `tools-basic` or `tools-platform` does — so those two are always-on. (`tools-platform` is kept whole rather than split further: its more sensitive members — triggers, notifications, agent-/skill-management — are accepted as always-on too; an Operator who cannot trust an Agent with its own workspace's domain tools should not grant that tool set at the Agent level, which remains the per-Agent control.)
- **A single coarse `apiVersion` requiring exact match against core.** Rejected — a core bump must never break in-the-wild plugins. Compatibility is **forward-compatible**: contracts evolve _append-only_ (new capability = optional method/field, never a new required member), and the manifest's `apiVersion` states the _minimum_ core API the plugin needs. Core rejects a plugin only if it needs a newer API than core provides (fixed by upgrading core, which the Operator controls) or if it targets a **dropped, long-deprecated major**. Core supports the current major **and one previous (N and N−1)**; a breaking change is a windowed major bump during which both run. `apiVersion` is a **positive integer** (majors only; there is no `v0`), so the oldest-supported floor is `1` while core is at major 1 — the N−1 slot opens only once core reaches major 2. A published `@platypuschat/plugin-sdk` is the compile-time contract and the single home of the Extension-point surface and `PLUGIN_API_VERSION`.
- **Skip-with-warning on a bad plugin at boot.** Rejected: a silently-skipped plugin means every Agent referencing its Contributions quietly loses tools — indistinguishable from data loss. Boot is **fail-loud, all-or-nothing** (a listed plugin that can't `import()`, has an invalid manifest, declares an incompatible `apiVersion`, or whose Plugin config fails schema validation aborts startup with a plugin-named error). Runtime Chat-turn resolution stays **graceful** (a Workspace pointing at a since-disabled backend degrades to an empty Tool set, as today). Strict at boot, forgiving at runtime — two moments, two postures.
- **Sanctioning raw DB / internal access from third-party Tool sets.** Rejected: with no isolation a third-party plugin _technically can_ reach internals, but the **supported** surface is the curated capability API the SDK exposes (scoped org/workspace/agent/user context plus vetted services — notifications, event bus, deploy-time credentials, Sandbox access). Raw Drizzle / internal query modules are **core-plugin-only** and unsupported for third parties (they couple to the schema and break across versions). This is a **contract/trust boundary, not an enforced sandbox** — the Operator vetting the plugin is the enforcement; the SDK's job is to make the supported path the easy one.

## Consequences

- **Loading.** Before the HTTP server starts, core loads the **always-on core set** (`@platypus/tools-basic`, `@platypus/tools-platform`) unconditionally, then reads `PLATYPUS_PLUGINS` (the **gate-able** list: the remaining core plugins `@platypus/web-fetch` / `@platypus/docker` and any third-party packages), `await import()`s each, validates and reads its manifest, and calls the internal registries per `contributes` entry — after the allowlist-based namespacing and boot-time collision check. Always-on plugins load first; a gate-able list entry that names an always-on plugin aborts boot (it is not an enable switch). `.env.example` / compose ship a sensible default gate-able list (`@platypus/web-fetch`); an empty list yields a deployment with **the always-on core tool sets only** — never zero tools. New core plugins in a later release are **not** auto-enabled (unless added to the always-on set) — the Operator adds them to their pinned list (explicit opt-in to new capability). Resolution differs by origin: **core plugins are internal backend modules** (`apps/backend/src/plugins/<name>/`) reached through a **static built-in map** — which _is_ the core allowlist, resolved from the trusted bundle, mirroring today's `import "./sandbox/backends/index.ts"` — so their `@platypus/*` names are logical ids, not published packages; **third-party plugins** are installed npm packages reached via dynamic `import()`. "Loaded identically" therefore means an identical manifest→register flow, not identical module resolution.
- **Manifest.** `export const plugin: PlatypusPlugin = { name, version, apiVersion, configSchema?, credentialsSchema?, contributes: { sandboxBackends?, toolSets?, ... } }`. `contributes` is keyed by Extension-point type (core-owned, fixed); each value is an array of the same registration payloads the internal `register*()` already accept. Adding an Extension point (the messaging gateway) is a new registry + new SDK type + new `contributes` key — purely additive, a minor API bump, disturbing no existing plugin.
- **Identity.** Plugin `name` and Contribution ids are globally unique and boot-checked; collision errors name the owning plugin (core now tracks id→plugin). Core ids stand alone; third-party ids are `${plugin.name}.${id}`. A third-party `name` is therefore the contribution-id prefix and **must be a short, url-safe slug** (lowercase letters, digits, hyphens), validated at boot — distinct from the npm package specifier the Operator lists (a package published as `@acme/platypus-widgets` sets `name: "widgets"`, ids `widgets.<id>`). Core identity is decided by the core-shipped allowlist of plugin names (not the scope string), so an impostor cannot claim the unprefixed namespace even by borrowing the `@platypus/*` name. A third-party plugin later absorbed into core sheds its prefix — a one-time, explicit migration for adopters.
- **Config & credentials.** Plugin config/credentials are deployment-wide, Operator-set, validated at boot, and injected into every Contribution's factory alongside the existing per-Workspace config. Per-Workspace Sandbox config/credentials and the ADR-0006 field-level authorization are unchanged.
  - **Amendment ([#312](https://github.com/willdady/platypus/issues/312)).** A Sandbox-backend Contribution's per-Workspace `configSchema` may additionally be a **factory** `(pluginConfig) => ZodSchema` of the plugin's resolved deploy-time config, which the loader resolves at load into a concrete schema before registering (so the static `configSchema.safeParse` consumers — save route, teardown, tool resolver — always receive a plain schema). This is append-only: a plain schema stays valid. It lets `@platypus/docker` fold its Operator network allowlist (`config.allowedNetworks`) into per-Workspace save-time validation, retiring the dedicated `PLATYPUS_SANDBOX_DOCKER_ALLOWED_NETWORKS` env var (ADR-0005).
- **Enablement** of a gate-able plugin is list membership; the always-on core set is not gated (listing it is a fail-loud misconfiguration). The per-backend `*_ENABLED` env gates of ADR-0003/0012 are retired in favour of listing `@platypus/docker` (and, when it lands, an SSH plugin).
- **Compatibility.** Extension-point contracts are append-only within a major; `SandboxBackend`'s five-tool core (ADR-0002) grows only via optional members. The SDK is published to public npm as `@platypuschat/plugin-sdk` (the `@platypuschat` npm org is owned by the project) so third-party authors can depend on it; core plugins are bundled/unpublished and consume the same types.
- **Observability.** A boot log enumerates loaded plugins, versions, and Contributions; a **read-only** `GET /plugins` API and a read-only Org-Admin "Installed plugins" view expose the same (no enable/disable over the API — deploy-time only). Existing catalogs (`GET /backends`, the Tools listing) are annotated with each Contribution's originating plugin.
- **Reframing built-ins.** The Docker backend and the ~11 native Tool sets become core plugins under `@platypus/*`, registering via `contributes` rather than direct `register*()` calls. Their ids are preserved, so no persisted reference changes.
- **Boundary vs MCP (non-goal #4).** "Talking over a protocol → MCP. Running code on the curated in-process SDK surface → Plugin. Raw internals → core only." A Plugin Tool set that merely proxies HTTP to an external service is a smell — that is MCP's role. Plugins extend Platypus's _own_ capabilities; they do not duplicate MCP.
