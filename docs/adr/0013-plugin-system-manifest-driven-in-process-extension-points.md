# Plugins are manifest-driven, in-process bundles that fill core-owned extension points

Platypus gains a first-class way to extend itself **without maintaining a fork**: a _Plugin_ is a distributable package — one version, one config namespace, one enable/disable switch — that the Operator installs at deploy time and that core loads **in-process** (no isolation). Each Plugin exports a typed **manifest** whose `contributes` block declares one or more **Contributions** to a fixed set of core-owned **Extension points** (initially Sandbox backends and Tool sets; a messaging-gateway adapter is planned as a third). Core reads the manifest and drives registration itself — the existing `registerSandboxBackend` / `registerToolSet` functions become core-internal, no longer the plugin-facing API. **Everything is a Plugin, including core built-ins**, which ship pre-bundled under the `@platypus/*` naming convention and flow through the identical `contributes` path. This settles the loading model, manifest format, and config mechanics the roadmap deferred to an ADR.

## Considered Options

- **Bare side-effect registration for external packages** (an out-of-tree package calls `registerSandboxBackend()` on import, as in-tree backends do today). Rejected: self-registration only works because core hard-codes `import "./sandbox/backends/index.ts"`; an external package can only self-register if _something_ imports it, and "no fork" forbids core growing an import line per plugin. The manifest inverts this — core discovers packages from an Operator-supplied list, `import()`s each, reads its manifest, and calls the internal `register*()`. Core stays ignorant of specific plugin names; the plugin author never calls `register*()`.
- **Pure-JSON `contributes` manifest (literal VS Code model).** Rejected: our Extension points need _code_ — a Sandbox backend ships a `create()` factory and two Zod schemas; a Tool set ships live `Tool` implementations. None survives as JSON. The manifest is a **typed TS object** exported from the package entry; only its top level (name / version / apiVersion / config namespace) is cheap declarative metadata. We borrow VS Code's `contributes` shape and its fixed contribution-point set — _not_ its marketplace, hot-loading, or sandboxed extension host.
- **A scanned `plugins/` directory** for discovery. Rejected in favour of an explicit Operator-supplied list (`PLATYPUS_PLUGINS`, promotable to a config file if it grows): the list _is_ the complete, auditable statement of what runs — read one value, know exactly what is active — and it doubles as the enable switch (below).
- **Flat, un-namespaced Contribution ids with boot-time collision detection.** Rejected _for third-party_: a third-party id could collide with a _future_ core built-in and break an existing deployment on upgrade. We take a **hybrid** — core Contributions keep flat ids (no data migration; every persisted `agent.toolSetIds` / `sandbox.backend` reference keeps working), while third-party Contributions are **auto-namespaced by plugin name** (`daytona.management`, `daytona.sandbox`). Authors write bare ids; core prefixes at load. The core-vs-third-party boundary is a **core-shipped allowlist of plugin names** (the packages this repo bundles) — _not_ the package-scope string, and _not_ list-membership (core plugins are listed too). `@platypus/*` is the naming convention for those packages, but the allowlist, resolved from the trusted bundle, is the authority: an Operator-installed `@platypus/impostor` is absent from the allowlist and is therefore treated as third-party (prefixed, curated-surface-only), so scope impersonation cannot smuggle a package in as core. This keeps the boundary — and the security model — independent of public-npm scope ownership.
- **Auto-namespace everything, including core.** Rejected: rewrites every stored id and forces a backfill across agents and sandboxes for no benefit the scope-based hybrid doesn't already deliver.
- **Per-workspace Plugin config, or Org-Admin-editable Plugin config in-app.** Rejected: contradicts "trust boundary is the deployment, no in-app install button." **Plugin config is deploy-time and Operator-owned**, keyed by plugin name (the "one config namespace"), validated at boot against a `configSchema` / `credentialsSchema` the manifest declares at the _plugin_ level. This is where a Plugin's **shared credential block** lives — one Daytona token injected into both its Sandbox backend and its management Tool set. The consequence — one credential block serves all tenants — is accepted; genuinely tenant-specific settings stay in the existing per-Workspace Sandbox row (ADR-0001) or an MCP. Plugin config is a _deployment_ layer above per-Workspace _resource_ config; the two layer, they do not merge (echoing the `adminEnv ⊃ userEnv` precedence of ADR-0004/0006).
- **A separate `_ENABLED` switch in addition to list membership.** Rejected: membership in `PLATYPUS_PLUGINS` _is_ the enable switch — one action, and disabling means removing the entry. Both changes need a backend restart anyway (deploy-time). This retires the per-backend env gates (`PLATYPUS_SANDBOX_DOCKER_ENABLED`, `PLATYPUS_SANDBOX_SSH_ENABLED` of ADR-0003/0012); Docker's opt-in posture falls out for free (just don't list it), so a manifest `enabledByDefault` field is unnecessary.
- **One core plugin per Contribution, or one monolithic core bundle.** Rejected both. Core built-ins are split by the rule **"a capability gets its own plugin when an Operator would plausibly want to deny it in isolation"** — so `@platypus/web-fetch` (network egress) and `@platypus/docker` (infra) stand alone, while pure utilities (`@platypus/tools-basic`: math, time) and Platypus-domain tools (`@platypus/tools-platform`: kanban, dashboards, triggers, memory, notifications, agent-discovery, skill-/agent-management) group by cohesion. Egress-in-isolation matters specifically for the sovereignty-focused deployment.
- **A single coarse `apiVersion` requiring exact match against core.** Rejected — a core bump must never break in-the-wild plugins. Compatibility is **forward-compatible**: contracts evolve _append-only_ (new capability = optional method/field, never a new required member), and the manifest's `apiVersion` states the _minimum_ core API the plugin needs. Core rejects a plugin only if it needs a newer API than core provides (fixed by upgrading core, which the Operator controls) or if it targets a **dropped, long-deprecated major**. Core supports the current major **and one previous (N and N−1)**; a breaking change is a windowed major bump during which both run. A published `@platypuschat/plugin-sdk` is the compile-time contract and the single home of the Extension-point surface and `PLUGIN_API_VERSION`.
- **Skip-with-warning on a bad plugin at boot.** Rejected: a silently-skipped plugin means every Agent referencing its Contributions quietly loses tools — indistinguishable from data loss. Boot is **fail-loud, all-or-nothing** (a listed plugin that can't `import()`, has an invalid manifest, declares an incompatible `apiVersion`, or whose Plugin config fails schema validation aborts startup with a plugin-named error). Runtime Chat-turn resolution stays **graceful** (a Workspace pointing at a since-disabled backend degrades to an empty Tool set, as today). Strict at boot, forgiving at runtime — two moments, two postures.
- **Sanctioning raw DB / internal access from third-party Tool sets.** Rejected: with no isolation a third-party plugin _technically can_ reach internals, but the **supported** surface is the curated capability API the SDK exposes (scoped org/workspace/agent/user context plus vetted services — notifications, event bus, deploy-time credentials, Sandbox access). Raw Drizzle / internal query modules are **core-plugin-only** and unsupported for third parties (they couple to the schema and break across versions). This is a **contract/trust boundary, not an enforced sandbox** — the Operator vetting the plugin is the enforcement; the SDK's job is to make the supported path the easy one.

## Consequences

- **Loading.** Before the HTTP server starts, core reads `PLATYPUS_PLUGINS` (a package list; core built-ins included, e.g. `@platypus/tools-basic,@platypus/web-fetch,@platypus/docker,...`), `await import()`s each, validates and reads its manifest, and calls the internal registries per `contributes` entry — after the allowlist-based namespacing and boot-time collision check. `.env.example` / compose ship a sensible default list; an empty list yields a deployment with **no Tools at all** (accepted, and self-documenting). New core plugins in a later release are **not** auto-enabled — the Operator adds them to their pinned list (explicit opt-in to new capability). Resolution differs by origin: **core plugins are internal backend modules** (`apps/backend/src/plugins/<name>/`) reached through a **static built-in map** — which _is_ the core allowlist, resolved from the trusted bundle, mirroring today's `import "./sandbox/backends/index.ts"` — so their `@platypus/*` names are logical ids, not published packages; **third-party plugins** are installed npm packages reached via dynamic `import()`. "Loaded identically" therefore means an identical manifest→register flow, not identical module resolution.
- **Manifest.** `export const plugin: PlatypusPlugin = { name, version, apiVersion, configSchema?, credentialsSchema?, contributes: { sandboxBackends?, toolSets?, ... } }`. `contributes` is keyed by Extension-point type (core-owned, fixed); each value is an array of the same registration payloads the internal `register*()` already accept. Adding an Extension point (the messaging gateway) is a new registry + new SDK type + new `contributes` key — purely additive, a minor API bump, disturbing no existing plugin.
- **Identity.** Plugin `name` and Contribution ids are globally unique and boot-checked; collision errors name the owning plugin (core now tracks id→plugin). Core ids stand alone; third-party ids are `${plugin.name}.${id}`. Core identity is decided by the core-shipped allowlist of plugin names (not the scope string), so an impostor cannot claim the unprefixed namespace even by borrowing the `@platypus/*` name. A third-party plugin later absorbed into core sheds its prefix — a one-time, explicit migration for adopters.
- **Config & credentials.** Plugin config/credentials are deployment-wide, Operator-set, validated at boot, and injected into every Contribution's factory alongside the existing per-Workspace config. Per-Workspace Sandbox config/credentials and the ADR-0006 field-level authorization are unchanged.
- **Enablement** is list membership; the per-backend `*_ENABLED` env gates of ADR-0003/0012 are retired in favour of listing `@platypus/docker` (and, when it lands, an SSH plugin).
- **Compatibility.** Extension-point contracts are append-only within a major; `SandboxBackend`'s five-tool core (ADR-0002) grows only via optional members. The SDK is published to public npm as `@platypuschat/plugin-sdk` (the `@platypuschat` npm org is owned by the project) so third-party authors can depend on it; core plugins are bundled/unpublished and consume the same types.
- **Observability.** A boot log enumerates loaded plugins, versions, and Contributions; a **read-only** `GET /plugins` API and a read-only Org-Admin "Installed plugins" view expose the same (no enable/disable over the API — deploy-time only). Existing catalogs (`GET /backends`, the Tools listing) are annotated with each Contribution's originating plugin.
- **Reframing built-ins.** The Docker backend and the ~11 native Tool sets become core plugins under `@platypus/*`, registering via `contributes` rather than direct `register*()` calls. Their ids are preserved, so no persisted reference changes.
- **Boundary vs MCP (non-goal #4).** "Talking over a protocol → MCP. Running code on the curated in-process SDK surface → Plugin. Raw internals → core only." A Plugin Tool set that merely proxies HTTP to an external service is a smell — that is MCP's role. Plugins extend Platypus's _own_ capabilities; they do not duplicate MCP.
